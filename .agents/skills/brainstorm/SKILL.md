# 头脑风暴 - 需求发现（面向 AI 编码增强）

在实现**之前**，引导 AI 协作完成需求发现，针对 AI 编码工作流优化：

* **任务优先**（立即记录想法）
* **先行动再提问**（减少低价值问题）
* **技术选型先调研**（避免让用户凭空想方案）
* **发散 → 收敛**（扩展思路，再锁定 MVP）
* **落在任务目录**（先稳定 `prd.md`，需要时再交给 `plan.md`）

---

## 何时使用

从 `$start` 触发，当用户描述开发任务时，尤其适用于：

* 需求不清或在变化
* 存在多种合理实现路径
* 需要权衡（体验、可靠性、可维护性、成本、性能）
* 用户可能一开始并不知道最佳选项

---

## 核心原则（不可妥协）

1. **任务优先（尽早记录）**
   开场即确保存在任务，用户的想法立刻有处可写。

2. **先行动再提问**
   若能从仓库代码、文档、配置、约定或快速调研推出答案——先做，再问。

3. **每条消息最多一个问题**
   不要用问题清单淹没用户。一次问一个，更新 PRD，再继续。

4. **优先给出具体选项**
   偏好/决策类问题，提供 2–3 个可行、具体的方案及权衡。

5. **技术选型先调研**
   若决策依赖行业惯例/同类工具/成熟模式，先调研再提方案。

6. **发散 → 收敛**
   初步理解后，主动考虑演进、相关场景、失败/边界情况——再收敛到带明确「不做范围」的 MVP。

7. **不要元问题**
   不要问「要不要我搜？」或「能否贴代码我好继续？」
   需要信息：自己搜/看。被阻塞：只问最小阻塞问题。

---

## 步骤 0：确保任务存在（始终）

在任何问答之前，确保任务已存在。若没有，立即创建一个。

* 用用户消息推导的**临时工作标题**。
* 标题不完美可以，稍后在 PRD 里 refine。

在 `.trellis/tasks/` 下手动创建任务目录。

目录至少包含：

- `prd.md`
- `notes.md`

模板参考：

- `.agents/skills/start/assets/prd-template.md`
- `.agents/skills/start/assets/notes-template.md`

立即用已知信息创建/填充 `prd.md`：

```markdown
# 头脑风暴：<简短目标>

## 目标

<一段话：做什么 + 为什么>

## 我已掌握的事实

* <来自用户消息的事实>
* <从仓库/文档发现的事实>

## 假设（临时）

* <待验证的假设>

## 待解决问题

* <仅保留阻塞类 / 偏好类；保持简短>

## 需求（演进中）

* <从已知开始写>

## 验收标准（演进中）

* [ ] <可测试的标准>

## 完成定义（团队质量线）

* 按需补充/更新测试（单测/集成）
* Lint / 类型检查 / CI 通过
* 若行为变化则更新文档/笔记
* 若有风险则考虑发布/回滚

## 明确不做（范围外）

* <本任务不做的事>

## 技术备注

* <已查看的文件、约束、链接、参考>
* <如有，调研摘要>
```

---

## 步骤 1：自动上下文（提问之前必做）

在问「代码长什么样？」这类问题之前，先自己收集上下文：

### 仓库检查清单

* 定位可能受影响的模块/文件
* 找既有模式（相似功能、约定、错误处理风格）
* 查看配置、脚本、已有命令定义
* 记下约束（运行时、依赖策略、构建工具）

### 文档检查清单

* 是否已有 PRD/spec/模板
* 是否有命令示例、README、ADR 等

把发现写入 PRD：

* 加入「我已掌握的事实」
* 约束/链接写入「技术备注」

---

## 步骤 2：划分复杂度（仍有参考价值，不阻塞建任务）

| 复杂度   | 判定                                               | 动作                                      |
| ------------ | ------------------------------------------------------ | ------------------------------------------- |
| **琐碎**  | 单行修复、错别字、显而易见的小改动                  | 跳过头脑风暴，直接实现         |
| **简单**   | 目标清晰，1–2 个文件，范围明确              | 问 1 个确认问题后实现      |
| **中等** | 多文件，略有模糊                         | 轻量头脑风暴（2–3 个高价值问题） |
| **复杂**  | 目标模糊、架构取舍、多路径 | 完整头脑风暴                             |

> 注：任务已在步骤 0 创建。分类只影响头脑风暴深度。

---

## 步骤 3：问题闸门（只问高价值问题）

在提出**任何**问题前，运行以下闸门：

### 闸门 A — 能否不靠用户推出？

若答案可通过以下方式得到：

* 仓库检查（代码/配置）
* 文档/spec/约定
* 快速市场/开源调研

→ **不要问。** 取回信息，摘要，更新 PRD。

### 闸门 B — 是否是元问题/偷懒问题？

例如：

* 「要不要我搜？」
* 「你贴一下代码我好继续？」
* 「代码长什么样？」（仓库可用时）

→ **不要问。** 采取行动。

### 闸门 C — 问题类型？

* **阻塞**：没有用户输入无法继续
* **偏好**：多种合理选择，取决于产品/体验/风险偏好
* **可推导**：应由检查/调研回答

→ 只问**阻塞**或**偏好**。

---

## 步骤 4：调研优先模式（技术选型必选）

### 触发条件（任一即调研优先）

* 任务涉及选型：方案、库、协议、框架、模板系统、插件机制或 CLI 交互惯例
* 用户问「最佳实践」「别人怎么做」「推荐」
* 用户无法合理枚举选项

### 调研步骤

1. 圈定 2–4 个可比工具/模式
2. 摘要常见惯例及存在原因
3. 映射到本仓库约束
4. 给出**2–3 个**在本项目可行的方案

### 调研输出格式（写入 PRD）

在 PRD 中增加一节（可放在技术备注下或独立）：

```markdown
## 调研备注

### 同类工具通常怎么做

* ...
* ...

### 来自本仓库/项目的约束

* ...

### 在本项目可行的方案

**方案 A：<名称>**（推荐）

* 工作方式：
* 优点：
* 缺点：

**方案 B：<名称>**

* 工作方式：
* 优点：
* 缺点：

**方案 C：<名称>**（可选）

* ...
```

然后只问**一个**偏好问题：

* 「你更倾向 A / B / C（或其他）？」

---

## 步骤 5：扩展扫视（发散）— 初步理解后必做

能概括目标后，在收敛前先主动拓宽思路。

### 扩展类别（每类 1–2 条要点即可）

1. **未来演进**

   * 1–3 个月后功能可能变成什么样？
   * 哪些扩展点现在值得预留？

2. **相关场景**

   * 哪些相邻命令/流程应与本功能保持一致？
   * 是否有对等预期（创建 vs 更新、导入 vs 导出等）？

3. **失败与边界**

   * 冲突、离线/网络失败、重试、幂等、兼容、回滚
   * 输入校验、安全边界、权限检查

### 扩展阶段给用户的消息模板

```markdown
我理解你想实现：<当前目标>。

在深入设计前，我先快速发散三类（减少返工）：

1. 未来演进：<1–2 条>
2. 相关场景：<1–2 条>
3. 失败/边界：<1–2 条>

本次 MVP 你希望包含哪些（或都不加）？

1. 仅当前需求（最小可行）
2. 增加 <X>（为后续扩展预留）
3. 增加 <Y>（增强稳健性/一致性）
4. 其他：请说明偏好
```

然后更新 PRD：

* MVP 内容 → `需求`
* 排除内容 → `明确不做`

---

## 步骤 6：问答循环（收敛）

### 规则

* 每条消息一个问题
* 可能时优先选择题
* 每次用户回答后：

  * 立即更新 PRD
  * 已答项从 `待解决问题` 移到 `需求`
  * 用可勾选项更新 `验收标准`
  * 澄清 `明确不做`

### 问题优先级（建议）

1. **MVP 范围边界**（包含/不包含什么）
2. **偏好决策**（在给出具体选项之后）
3. **失败/边界行为**（仅对 MVP 关键路径）
4. **成功指标与验收标准**（如何证明可用）

### 偏好问题格式（选择题，建议）

```markdown
关于 <主题>，你更倾向哪种？

1. **选项 A** — <含义 + 权衡>
2. **选项 B** — <含义 + 权衡>
3. **选项 C** — <含义 + 权衡>
4. **其他** — 请说明偏好
```

---

## 步骤 7：提出方案并记录决策（复杂任务）

需求足够清晰后，若尚未通过「调研优先」给出方案，则提出 2–3 种方案：

```markdown
根据当前信息，这里有 2–3 种可行方案：

**方案 A：<名称>**（推荐）

* 做法：
* 优点：
* 缺点：

**方案 B：<名称>**

* 做法：
* 优点：
* 缺点：

你更倾向哪个方向？
```

在 PRD 中以轻量 ADR 形式记录结果：

```markdown
## 决策（轻量 ADR）

**背景**：为何需要此决策  
**决定**：选择了哪种方案  
**后果**：权衡、风险、可能的后续改进
```

---

## 步骤 8：最终确认 + 实施计划

待解决问题解决后，用结构化摘要确认完整需求：

### 最终确认格式

```markdown
以下是我对完整需求的理解：

**目标**：<一句话>

**需求**：

* ...
* ...

**验收标准**：

* [ ] ...
* [ ] ...

**完成定义**：

* ...

**明确不做**：

* ...

**技术方案**：
<简要摘要 + 关键决策>

**实施计划（小步 PR）**：

* PR1：<脚手架 + 测试 + 最小贯通>
* PR2：<核心行为>
* PR3：<边界 + 文档 + 清理>

以上是否正确？若可以，我将开始实现。
```

### 子任务拆分（复杂任务）

对多个相互独立工作项，可在 `.trellis/tasks/` 下手动创建子任务目录。

---

## PRD 目标结构（终稿）

`prd.md` 应收敛为：

```markdown
# <任务标题>

## 目标

<为什么 + 做什么>

## 需求

* ...

## 验收标准

* [ ] ...

## 完成定义

* ...

## 技术方案

<关键设计与决策>

## 决策（轻量 ADR）

背景 / 决定 / 后果

## 明确不做

* ...

## 技术备注

<约束、参考、文件、调研笔记>
```

---

## 反模式（坚决避免）

* 向用户要能从仓库推出的代码/上下文
* 在给出具体选项前让用户选方案
* 关于是否调研的元问题
* 只盯着最初一句需求，不考虑演进与边界
* 头脑风暴飘着跑却不更新 PRD

---

## 与 Start 工作流的衔接

头脑风暴完成（步骤 8 确认通过）后，流程进入任务工作流的**阶段 2：准备实现**：

```text
头脑风暴
  步骤 0：创建任务目录 + 填充 PRD
  步骤 1–7：发现需求、调研、收敛
  步骤 8：最终确认 → 用户批准
  ↓
任务工作流阶段 2（准备实现）
  代码-Spec 深度检查（如适用）
  → 调研代码库（基于已确认的 PRD）
  → 直接阅读相关 spec
  → 在任务目录内继续
  ↓
任务工作流阶段 3（执行）
  实现 → 检查 → 收尾
```

任务目录与 PRD 已在头脑风暴中创建。

若任务需要正式实施计划，下一步为：

- `$writing-plans`

计划须写入：

- `.trellis/tasks/<task>/plan.md`
