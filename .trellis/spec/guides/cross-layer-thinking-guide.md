# 跨层思考指南

> **目的**：实现前先理清跨层数据流。

---

## 问题

**大多数缺陷发生在层与层边界**，而非层内部。

常见跨层问题：
- API 返回格式 A，前端期望格式 B
- 数据库存 X，服务转成 Y 但丢失信息
- 多层以不同方式实现同一逻辑

---

## 实现跨层功能之前

### 步骤 1：绘制数据流

画出数据如何流动：

```
来源 → 转换 → 存储 → 读取 → 转换 → 展示
```

对每一步自问：
- 数据是什么格式？
- 可能出什么问题？
- 谁负责校验？

### 步骤 2：识别边界

| 边界 | 常见问题 |
|----------|---------------|
| API ↔ 服务 | 类型不匹配、缺字段 |
| 服务 ↔ 数据库 | 格式转换、null 处理 |
| 后端 ↔ 前端 | 序列化、日期格式 |
| 组件 ↔ 组件 | props 形状变更 |

### 步骤 3：定义契约

对每个边界：
- 输入格式确切是什么？
- 输出格式确切是什么？
- 可能出现哪些错误？

---

## 常见跨层错误

### 错误 1：隐式格式假设

**不好**：未核对就假设日期格式

**好**：在边界处显式做格式转换

### 错误 2：校验分散

**不好**：在多层重复校验同一件事

**好**：在入口处只校验一次

### 错误 3：抽象泄漏

**不好**：组件直接了解数据库 schema

**好**：每层只了解相邻层

---

## 跨层功能检查清单

实现前：
- [ ] 已绘制完整数据流
- [ ] 已识别所有层边界
- [ ] 已定义每层边界的格式
- [ ] 已确定校验发生的位置

实现后：
- [ ] 已用边界情况测试（null、空、非法）
- [ ] 已在各边界验证错误处理
- [ ] 已确认数据能完整往返

---

## 何时编写流程文档

在以下情况编写详细流程文档：
- 功能跨越 3 层以上
- 涉及多个团队
- 数据格式复杂
- 该功能曾引发缺陷
