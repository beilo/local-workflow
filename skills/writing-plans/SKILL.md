# 编写计划

在 `prd.md` 已清晰、且尚未开始实现时使用本技能。

## 输出位置

将实施计划写入：

- `.trellis/tasks/<task>/plan.md`

从此模板开始：

- `assets/plan-template.md`

## 输入

阅读：

- 任务的 `prd.md`
- 若存在，任务的 `notes.md`
- `.trellis/spec/` 下相关文件
- 你计划遵循的既有代码模式

## 目标

把当前任务变成简短、具体、可执行的计划。

计划应明确：

- 将修改哪些文件
- 工作顺序
- 每段完成后要验证什么
- 风险点在哪里

## 规则

### 若任务简单

短小的 `plan.md` 即可。

### 若任务为中大型

写更完整的 `plan.md`，包含：

- 阶段
- 涉及文件
- 检查点
- 风险部分的回退说明

## 必备章节

`plan.md` 应包含：

1. 目标
2. 范围
3. 涉及文件
4. 执行步骤
5. 验证点
6. 风险 / 备注

## 计划质量线

每个执行步骤至少要交代：

- 涉及文件
- 完成标志
- 验证方式
- 若风险较高，写明停止条件或回退思路

执行者读完计划后，应能直接开工，而不是边做边猜。

## 禁止占位写法

以下内容一旦出现在 `plan.md` 中，视为计划质量不达标：

- `TBD`、`TODO`、`后续补充`、`实现时再看`
- 「补充异常处理」「按现有模式处理」「补齐边界情况」这类没有具体动作的空话
- 「写测试」但没有说明验证对象、验证方式或覆盖范围
- 「类似上一步」「参考前文」这类要求执行者自己回推上下文的描述
- 引用了尚未定义的文件、模块、函数、类型或命令

## 自审（内联）

写完整份 `plan.md` 后，回头对照 `prd.md` 做一次快速自审，并直接修正文档：

1. **PRD 覆盖检查**
   - `prd.md` 里的核心需求、验收标准、明确不做，是否都在计划中得到对应？
   - 若某条需求本轮不做，应在 `plan.md` 中明确写成范围外，而不是默认忽略。
2. **占位符扫描**
   - 搜索整份计划，清掉「TODO」「后续补充」「按需处理」等占位写法。
3. **命名 / 类型 / 路径一致性**
   - 后文提到的文件路径、模块名、函数名、命令名，是否与前文保持一致？
   - 不要前面写 `plan.md`，后面又变成 `planning.md`；不要前面说改 A 文件，后面步骤却落到 B 文件。
4. **可执行性检查**
   - 执行者是否可以只看这份计划就知道先做什么、做到什么算完成、如何验证？
   - 若某一步仍需要大量临场猜测，说明计划还不够具体，应继续补齐。

若发现问题，直接原地修正，不额外开启评审回路。

## 执行交接

保存 `plan.md` 后，不要直接开始写代码，先根据任务复杂度给出执行建议并征求用户选择：

```markdown
计划已写入 `<task>/plan.md`。接下来有 3 种执行方式：

1. **默认轻量单会话实现** —— 简单任务推荐，直接在当前会话推进
2. **$executing-plans** —— 按计划分步执行，并在检查点停下来复核
3. **$subagent-driven-development** —— 仅在任务可清晰拆分时使用

你希望按哪种方式继续？
```

建议原则：

- 简单任务：优先默认轻量单会话实现，或使用 `$executing-plans`
- 多步骤但仍需主会话持续把控：优先 `$executing-plans`
- 只有在任务能清晰拆分、边界明确时，才使用 `$subagent-driven-development`

## 与执行的关系

- `brainstorm` 在 `prd.md` 中稳定任务
- `writing-plans` 将其转为 `plan.md`
- 默认轻量单会话实现、`executing-plans` 或 `subagent-driven-development` 随后可使用：
  - `plan.md`
  - 简单的 `prd.md`
  - 用户明确提供的其他任务文档

## 边界

此处不要开始写代码。

本技能只编写或更新 `plan.md`。
