# 执行计划

这是可选的执行类技能。

不属于默认工作流。

仅当用户明确要求在当前会话中按文档驱动执行时使用。

## 可接受的输入

本技能可从以下任一文档执行：

- `plan.md`
- `prd.md`
- 用户明确拖入或指向的任务文档

## 入口判断

开始执行前，先判断输入是否真的适合直接落代码：

1. **若已有 `plan.md`**
   - 优先按 `plan.md` 执行，不要绕过计划自己重构步骤。
2. **若只有 `prd.md`**
   - 先确认这份 PRD 已完成基本收敛：
     - 关键目标清楚
     - 范围边界明确
     - 用户已确认书面需求，或当前任务确实足够简单可直接执行
3. **若文档仍有明显占位符或歧义**
   - 先停下来补文档，不要边猜边写代码。

## 文档处理规则

### 若文档为 `plan.md`

直接使用。

但在开工前先快速检查：

- 是否还残留 `TODO`、`待补充`、`按需处理` 之类占位写法
- 是否写清每一步的涉及文件、完成标志、验证方式
- 若发现计划已经无法直接执行，先回写 `plan.md`，再继续

### 若文档为其他已可直接执行的任务文档

则从中直接执行。

### 若文档为简单的 `prd.md`

可从该文档推导最小执行清单并直接执行。

不必强行先单独写 `plan.md`。

但要先在 `notes.md` 中记下：

- 本次执行方式
- 来源文档
- 你准备采用的最小执行顺序

### 若文档存在明显缺口

以下情况不允许直接执行：

- 目标看似明确，但范围仍在飘
- 验收标准缺失，导致完成与否无法判断
- 涉及多个独立子系统，却没有拆分
- 需要临场做新的设计决策，文档里却没有依据

出现这些情况时，先回到 `brainstorm` 或 `writing-plans`，不要把执行会话变成需求会话。

### 若文档为复杂的 `prd.md`

不要猜测。

停止直接执行，先使用本地 `writing-plans` 工作流创建：

- `.trellis/tasks/<task>/plan.md`

## 执行边界

若仅阅读并分类文档，仍属分析阶段。

若即将开始改代码，必须先做等价于：

- `$before-dev`

代码改动结束后，必须再做等价于：

- `$check-dev`

然后继续：

- `$finish-work`
- 人工测试并提交后，更新 `notes.md` 中的最终状态与交接信息

## 执行过程记录

在执行过程中，持续把关键状态写入 `notes.md`：

- **当前状态**：现在在做什么
- **已完成**：本轮已经落地了什么
- **下一步**：下一步准备做什么
- **已完成验证**：本轮验证依据是什么
- **计划偏差与回写**：若实际实现偏离 `prd.md` / `plan.md`，必须记下偏差，并说明是否已经回写文档

对简单任务，至少在以下时机更新一次：

1. 开始编码前
2. 关键步骤完成后
3. 进入 `check-dev` / `finish-work` 前

## 停止条件

执行中若出现以下任一情况，应先停下，不要硬着头皮写：

- 文档与代码现实明显不一致
- 需要新增或推翻原有设计决策
- 发现跨层影响比预期大很多
- 计划中的验证方式已经不成立
- 出现新的阻塞问题，当前文档无法指导如何处理

处理原则：

- **小偏差**：先记入 `notes.md`，必要时补一两处文档后继续
- **大偏差**：回写 `prd.md` / `plan.md`，必要时重新走 `brainstorm` 或 `writing-plans`

## 推荐流程

1. 阅读所提供的任务文档
2. 判断其属于：
   - 可直接执行的 `plan.md`
   - 简单 `prd.md`
   - 复杂 `prd.md`
   - 存在缺口、应先回写的文档
3. 若将开始执行，运行 `before-dev`
4. 在 `notes.md` 中记录执行方式、来源文档、当前检查点
5. 按文档执行，并在检查点持续更新 `notes.md`
6. 若发生明显偏差，先停下并回写文档
7. 运行 `check-dev`
8. 运行 `finish-work`
