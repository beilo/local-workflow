# 子代理驱动开发

这是可选的执行类技能。

不属于默认工作流。

仅当以下情况使用：

- 用户明确要求子代理执行，或
- 任务可清晰拆成相互独立的块

## 适用判断

满足以下条件时，才值得进入子代理模式：

- 可以拆成**边界清楚**的子任务
- 各子任务的主要改动文件尽量不重叠
- 主会话能够提前定义每块的目标、输入和完成标准
- 主会话有能力负责最终集成、冲突处理与质量把关

以下情况通常**不适合**：

- 只是简单 `prd.md`
- 需求仍模糊，还需要边做边定
- 多个子任务会频繁改同一批文件
- 需要主会话持续盯着一个线性调试过程

若不适合，退回：

- 默认轻量单会话实现，或
- `$executing-plans`

## 可接受的输入

本技能可从以下任一文档执行：

- `plan.md`
- `prd.md`
- 用户明确拖入或指向的任务文档

## 优先级规则

若 `plan.md` 已存在，优先按 `plan.md` 拆分与分派。

不要绕过计划，直接把模糊目标扔给子代理。

## 文档处理规则

### 若文档为 `plan.md`

直接使用。

### 若文档为简单的 `prd.md`

优先不用子代理。

简单 PRD 通常应留在主会话，使用 `executing-plans` 或直接实现。

### 若文档为结构化但复杂的 `prd.md`

可先拆成明确子任务，再分派子代理。

若仍过于模糊，先停并转为 `plan.md`。

### 若文档存在明显缺口

以下情况先不要派发：

- 关键术语未定义
- 验收标准不足
- 文件边界尚未想清
- 子任务之间的依赖顺序不明确

先由主会话补齐文档，再进入子代理执行。

## 前后规则

任何子代理开始写代码之前：

- 控制方必须先做等价于 `$before-dev` 的步骤
- 每个子代理须收到相关任务上下文与 spec 路径
- 控制方须先在 `notes.md` 中记下本次执行方式与分工策略

代码完成后：

- 对整个任务做 `check-dev`
- 然后继续 `finish-work`
- 人工测试并提交后再更新 `notes.md` 中的最终状态与交接信息

## 控制方职责

主会话不是“发单器”，而是整个执行过程的控制方。必须负责：

- 判断任务是否真的适合子代理
- 预先拆分子任务，避免职责重叠
- 为每个子代理指定文件边界或责任范围
- 处理子代理之间的冲突、依赖与集成顺序
- 汇总验证结果
- 在 `notes.md` 中记录分工、偏差和最终结论

## 子代理分派要求

每个子代理至少要收到以下信息：

- 任务范围：本子任务做什么，不做什么
- 相关文档：`prd.md` / `plan.md` / spec 路径
- 文件所有权：优先说明它主要负责哪些文件或模块
- 完成标准：做到什么算完成
- 验证要求：至少要跑什么验证

并明确提醒：

- 不要擅自扩大范围
- 不要回滚主会话或其他子代理已完成的改动
- 若发现文档与现实不一致，先回报，不要自行改写任务目标

## 子代理回报格式

要求子代理返回的内容至少包含：

- 改动了哪些文件
- 做了哪些验证
- 遇到了哪些阻塞或风险
- 是否偏离原计划
- 哪些后续工作仍需主会话处理

主会话整合后，再统一写入 `notes.md`。

## 回退规则

执行中若发现以下情况，应停止继续扩散子代理：

- 子任务边界开始互相重叠
- 需要频繁修改同一文件
- 新问题要求重新设计整体方案
- 集成成本已经高于单会话执行收益

此时应回退到：

- 主会话继续收口，或
- 改用 `$executing-plans`

## 推荐流程

1. 阅读所提供的任务文档
2. 判断是否适合子代理执行
3. 若适合，执行 `before-dev`
4. 在 `notes.md` 中记录执行方式、来源文档、分工策略
5. 分派子代理，附带：
   - 任务范围
   - 涉及文件 / 文件所有权
   - 相关 spec 文件
   - 完成标准与验证要求
6. 收集回报并整合结果
7. 若发现边界失控，立即回退到主会话或 `$executing-plans`
8. 对整个任务运行 `check-dev`
9. 运行 `finish-work`
