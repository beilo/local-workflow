# 代码复用思考指南

> **目的**：在写新代码之前先停下来想一想——是否已存在？

---

## 问题

**重复代码是不一致缺陷的第一大来源。**

当你复制粘贴或重写已有逻辑时：
- 缺陷修复无法同步到所有副本
- 行为随时间分化
- 代码库更难理解

---

## 写新代码之前

### 步骤 1：先搜索

```bash
# 搜索相似的函数名
grep -r "functionName" .

# 搜索相似的逻辑
grep -r "keyword" .
```

### 步骤 2：自问这些问题

| 问题 | 若答案为是… |
|----------|-----------|
| 是否已有类似函数？ | 使用或扩展它 |
| 别处是否也在用该模式？ | 遵循既有模式 |
| 这是否可做成共享工具？ | 在合适位置创建 |
| 是否正在从另一文件复制代码？ | **停**——抽到共享处 |

---

## 常见重复模式

### 模式 1：复制粘贴函数

**不好**：把校验函数复制到另一文件

**好**：抽到共享工具，在需要处 import

### 模式 2：相似组件

**不好**：新建与现有组件 80% 相似的新组件

**好**：用 props/变体扩展现有组件

### 模式 3：重复常量

**不好**：在多个文件定义同一常量

**好**：单一事实来源，各处引用

---

## 何时抽象

**适合抽象时**：
- 相同代码出现 3 次以上
- 逻辑复杂到容易出 bug
- 多人可能都需要这段逻辑

**不宜抽象时**：
- 只用一次
- 微不足道的一行
- 抽象比重复更复杂

---

## 批量修改之后

若你对多个文件做了相似修改：

1. **回顾**：是否遗漏实例？
2. **搜索**：用 grep 查找遗漏
3. **考虑**：是否应抽象？

---

## 陷阱：不对称机制产生相同输出

**问题**：当两种不同机制必须产生同一组文件（例如初始化的递归目录复制 vs. 更新时手动的 `files.set()`），结构性变更（重命名、移动、新增子目录）只会通过自动机制传播，手动路径会悄悄漂移。

**现象**：初始化完美，但更新时文件路径错误或完全遗漏。

**预防检查清单**：
- [ ] 迁移目录结构时，搜索**所有**引用旧结构的代码路径
- [ ] 若一路径由 glob/复制推导，另一路径为手动列举，则手动路径也需要更新
- [ ] 增加回归测试，比较两种机制的输出

---

## 提交前检查清单

- [ ] 已搜索是否存在类似代码
- [ ] 没有本应共享却复制粘贴的逻辑
- [ ] 常量在单一位置定义
- [ ] 相似模式遵循相同结构
